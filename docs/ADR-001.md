## Context

The project is pivoting from a custom SQL engine and a custom append-only
journal to a **typed embedded control-plane state kernel**. The kernel's job is
to give a single-process application a durable, inspectable source of truth for
its control-plane state.

## Semantic value being preserved

The pivot changes the storage substrate, not the semantics. The kernel must
continue to provide:

1. **Events** — an append-only, ordered log of typed domain events as the
   canonical record of what happened.
2. **Projections** — derived read models rebuilt deterministically from the
   event log.
3. **Durable jobs** — queued work that survives process restarts, with an
   explicit state machine and an effect-mode contract: reconciliation-required
   jobs (the default) become `Uncertain` on lease expiry and are never
   silently retried; only explicitly idempotent jobs re-enter the retry path.
4. **Honest uncertainty** — the system never pretends to know more than it
   does: indeterminate commits and indeterminate claims return only a
   transaction ID (never executable work), and typed recovery APIs resolve
   them as committed, absent, or still indeterminate.

## Decision

**SQLite is the production storage substrate.** All events, projections, and
job state are stored in a single SQLite database file owned by one process.

## Why the custom journal is not the production default

- Correct crash-safe storage (fsync ordering, torn-write handling, checksums,
  recovery) is a multi-year hardening effort; the v1 journal had none of that
  maturity.
- It duplicated capabilities SQLite already provides (atomic commits, WAL,
  integrity checking) with a strictly worse test surface.
- Maintenance cost crowded out work on the kernel's actual value: typed
  events, projections, and jobs.
- The journal remains available for study at `archive/append-only-journal-v1`
  / tag `journal-v1-experimental`.

## Why SQLite

- Battle-tested durability and crash recovery (WAL mode, `PRAGMA synchronous`).
- Single-file, zero-ops, embedded — matches the single-writer-process model.
- Transactions give atomic event append + projection update + job transition.
- Ubiquitous tooling for inspection, backup, and debugging.
- Well-understood performance envelope for control-plane (not data-plane) load.

## Why redb remains a future possibility

redb is a pure-Rust embedded ACID key-value store. It could remove the C
dependency and simplify cross-compilation (WASM, embedded targets). It is not
chosen now because its ecosystem maturity, tooling, and long-term format
stability are behind SQLite's, and the kernel does not yet have evidence that
SQLite is a limiting factor.

## Why a public backend abstraction is deferred

Designing a backend trait before a second backend exists produces speculative,
leaky abstractions. The storage layer is kept as a private internal module with
a narrow surface; if/when a second backend is warranted, the abstraction is
extracted from two working implementations rather than invented up front.

## Evidence that would justify revisiting

- Deployment targets where linking C SQLite is infeasible (e.g. strict WASM or
  embedded environments) become a real requirement.
- Measured SQLite performance or file-size limits on representative
  control-plane workloads.
- A concrete second backend (e.g. redb) passing the full kernel invariant test
  suite, demonstrating the abstraction cost is paid for.
- Operational incidents traceable to SQLite behavior that a different substrate
  demonstrably avoids.
