# minisqlite-node (spike)

napi-rs bindings for the minisqlite control-plane kernel, targeting Node and
Bun (first consumer: the Synara Electron app).

Status: exploratory spike. The binding surface is intentionally small and the
crate is not published.

## Surface

- `Store.open(path, durability?)`, `store.close()`
- `store.commit(batch)` — events, projection patches, job enqueues, and job
  acks in one atomic batch
- `store.claimJobs(request)`, `store.extendLease(...)`
- `store.recoverClaim(transactionId, nowMs)`, `store.recoverTransaction(transactionId)`
- Reads: `jobs`, `projectionGet`, `projectionVersion`, `streamVersion`, `eventsAfter`
- `newId()` — generate IDs JS-side so indeterminate outcomes are recoverable

Conventions: IDs are 32-char lower-case hex strings; byte payloads are
Buffers; Rust error enums throw JS `Error`s whose `code` property names the
variant (`Conflict`, `Validation`, `Storage`, `CommitIndeterminate`,
`ClaimIndeterminate`, `LeaseConflict`, ...). Indeterminate errors embed
`transactionId=<hex>` in the message for recovery.

## Build and test

```sh
bun install
bunx napi build --platform   # or: bun run build (release)
node examples/synara_provider_turn.mjs
bun examples/synara_provider_turn.mjs
bun test
```

`index.js` / `index.d.ts` are generated by `napi build` and committed; the
platform `.node` binary is not.

## Known next steps

- Electron: the addon should load as-is in Electron's Node (same N-API ABI),
  but an `electron-rebuild`-free packaging flow and a smoke test inside the
  Synara desktop app are still to be verified.
- Prebuilt binaries per platform (napi-rs GitHub Actions matrix) before any
  real consumer adopts this.
- Richer error payloads: attach `transactionId` as a structured property on
  indeterminate errors instead of embedding it in the message.
- Remaining API surface: fail/cancel/resolve job operations exist on
  `commit` batches in Rust but are not yet exposed; `projectionScan*`,
  `jobsPage`, `stats`, `verify`, `backup` are unbound.
